# Sprint 2.3 — pd2-tools integration

**Branch:** `feature/meta-build-aggregator` on `314159DD/pd2-tools` (fork)
**PR:** [coleestrin/pd2-tools#20](https://github.com/coleestrin/pd2-tools/pull/20)
**Status:** Shipped to PR 2026-05-11; awaiting upstream review.

**Goal:** Port the aggregator's filter and aggregate layers into `coleestrin/pd2-tools` as a new `/meta` page, wired directly to their Postgres so the whole rare/magic/crafted population is in scope instead of our 1500-character sample.

This sprint collapsed what the integration-notes doc estimated as 4 sub-sprints (port logic, port UI, wire DB, launch) into one push because the upstream stack already runs React + Mantine, so styling adaptation was the only large fork in the road.

---

## What landed

### Backend (`api/`)
- `routes/meta.ts` — single GET endpoint with strict query-param parsing, `autoCache(900)`, `validateSeason` middleware.
- `database/postgres/meta.ts` — 7 aggregation methods on `MetaDB_Postgres`:
  - `findCohort` (className + level + gameMode + skills filter)
  - `aggregateItemUsage` (Unique/Set/Runeword)
  - `aggregateSkillUsageClassified` (prereq vs build, `numAtTwenty` threshold)
  - `aggregateMercType` + `aggregateMercItems`
  - `aggregateLevelDistribution`
  - `aggregateAffixMods` (port of `aggregate/affixMods.ts`)
  - `aggregateAvgStats` (cohort averages of str/dex/vit/energy + life + mana from `character.attributes`; resistances skipped on purpose since they cap at 75)
- `types/meta.ts` — request and response types, `IClassifiedSkillRow`, `IAvgStatRow`, `IAffixModRow`.
- `routes/meta.test.ts` — 67 Jest + supertest tests covering validation paths and parity vs the aggregator's fixture corpus.

### Frontend (`web/`)
- `pages/Meta.tsx` — top-level page, centered "Meta" title with BETA pill, welcome banner with Discord callout for feedback.
- `components/meta/` — 11 components:
  - Ported: FilterForm, BuildSheet, ItemFrequencyTable, AffixFrequencyTable, CharmPanel (placeholder), DataFreshness, MatchBanner, DiffView.
  - New: AvgStatsPanel, CollapsibleSection, TopAffixAveragesPanel.
- `data/skill-classification.json` — 220+ skills tagged Core or Synergy. Generated by `scripts/build-skill-classification.py` (wiki tree → role, with hand-curated overrides and `receivesBonusesFrom` fallback for mixed trees like Offensive Auras).
- Plus the rest of the standalone's pure data: `skill-prereqs.json`, `builds.json`, `mod-dictionary.json`, `item-slots.json`.

### UX iteration (driven by Steven's clicktrough)
- Compact skill picker (220 px scroll, line-on-line rows, 6 rows visible).
- Class buttons stay on one line at any viewport via 3-tier responsive labels (Sorceress / Sorc / S).
- Top items + Affix patterns tables use fixed column widths so columns don't jump between slot tabs.
- Affix patterns gained an "All slots" tab; `TopAffixAveragesPanel` reuses that aggregation in the Average build stats section, with resists and skill/proc mods filtered out since they don't aggregate meaningfully.
- Collapsible sections everywhere (Core Skills, Level distribution, Top items, Affix patterns, Charms, Average build stats) via the new `CollapsibleSection` wrapper.
- Core Skills uses the new classification to default-hide synergies and prereqs behind toggles. Single popularity-sorted list with a Type badge per row (Main rows get no badge, Synergy + Prereq get gray pills).

### Tooling
- `scripts/build-skill-classification.py` — wiki scraper for the new classification JSON. Rerun on PD2 patches.

## What's deferred

- **Charm aggregation** — backend route + parsing not built. Frontend renders a placeholder card. Same situation as in the standalone before its Phase 2.
- **Average build stats: resistances / FCR / FHR / etc.** — pulled from `character.attributes` + `life` + `mana` only. Computing the gear-summed-then-capped values would mean replicating PD2's cap and breakpoint logic server-side. Static note in the panel calls this out.
- **Per-character synergy classification on the backend.** Right now classification is class-static (from the wiki JSON). A future iteration could classify per-character against `receivesBonusesFrom` and return `numAsCore` / `numAsSynergy` alongside `numAsBuild` / `numAsPrereq`. Static is good enough for most builds; per-character is better for builds where the same skill swaps role (Sacrifice in Zealot vs Sacrifice builds).
- **Tier 5 re-rank / direct Anthropic plumbing** — not relevant here, that's a JobFlowAI line item.
