# Sprint 2.2 — Correctness audit + integration prep

**Date:** 2026-05-11
**Owner:** Steven Obst
**Status:** Approved (design)

## Context

After Sprint 2.1 launched on Reddit, two correctness bugs slipped past unit tests and were caught only by users comparing our output to pd2.tools' UI:

- **Sprint 2.1.1** — server stats endpoints were never being skill-filtered. `Top items by slot`, `Mercenary`, `Level distribution` all aggregated across the entire class instead of the build cohort. Bug pre-dated the MVP — nobody had noticed.
- **Sprint 2.1.2** — `data/item-slots.json` (regenerated in 2.1 from a 250-character snapshot) was silently dropping ~30% of named items per cohort, including entire popular sets (M'avina), popular runewords (Lore, Hustle), and popular uniques (Hellrack, Kuko Shakaku).

Both shared a shape: *the API returns N items, our display shows M, N ≠ M, no automated check noticed.* This sprint installs that check.

Meanwhile, coleestrin (pd2.tools maintainer) responded to our launch with an integration offer:

> *"That seems extremely useful! Nice work. If you wanted to make it a subpage on pd2.tools that would be awesome, I'm completely open to that. It's open source at https://github.com/coleestrin/pd2-tools, you could contribute it there, what do you think?"*

Sprint 2.2 takes the "defer + prepare" stance: get our correctness story bulletproof (which we'd need either way) and lay the groundwork for an integration conversation, without committing to a port yet.

## Goals

1. **Validation test suite** that would have caught both 2.1.1 and 2.1.2 before they shipped, and that catches the same class of bug in the future.
2. **Open-source posture** prepared for eventual contribution to `coleestrin/pd2-tools`: license file, polished README, clean attribution.
3. **Integration-prep doc** to use as the agenda when we talk to coleestrin about how a port would actually work — not a port plan, a "things to align on before porting" plan.

## Non-goals

- No actual porting work to pd2-tools. That decision is deferred until after this sprint.
- No new product features. The build presets, prereq classifier, etc. shipped in 2.1 are the feature set for now.
- No live-API tests in CI. Tests use captured fixtures (deterministic, fast, no rate-limit risk).
- No UI changes — pure infrastructure sprint.

### Fixture shape

Each fixture captures the full set of server endpoints we depend on, in a single JSON file:

```json
{
  "_meta": {
    "build": "amazon-lightning-fury",
    "fetchedAt": "2026-05-11T13:42:11Z",
    "filter": { "gameMode": "hardcore", "className": "Amazon", "minLevel": 80 },
    "skills": [{ "name": "Lightning Fury", "minLevel": 20 }]
  },
  "itemUsage": [ ... ],          // /characters/stats/item-usage
  "skillUsage": [ ... ],         // /characters/stats/skill-usage
  "mercTypeUsage": [ ... ],      // /characters/stats/merc-type-usage
  "mercItemUsage": [ ... ],      // /characters/stats/merc-item-usage
  "levelDistribution": { ... }   // /characters/stats/level-distribution
}
```

One fetch per endpoint per build = 7 builds × 5 endpoints = 35 HTTP calls per refresh. The refresh script runs all 7 builds with concurrency-limited fetches.

### Refresh script

`scripts/refresh-validation-fixtures.ts`:

- Reads canonical builds list (hardcoded array referencing names from `data/builds.json`).
- For each build, calls all 5 stats endpoints via the same `src/lib/api.ts` functions the app uses — **deliberately** so any bug in the URL builder affects the fixture exactly the same way it affects production. If our API client has a bug, fixtures capture that bug, tests pass, but humans comparing fixtures to pd2.tools' UI will spot it.
- Writes each build's combined response to `tests/fixtures/validation/<build>.json`.
- Concurrency capped at 3 (courtesy to coleestrin's API).
- Total runtime target: under 30 seconds for all 7 builds.

Re-run cadence: weekly during active development; on every PD2 patch; whenever a parity test starts looking suspicious.

### Test assertions

`tests/validation/parity.test.ts` — one `describe` block per build, each running these assertions:

**1. totalSample consistency across endpoints**
The server reports the same `totalSample` value on every stats endpoint for the same cohort. Assert that fixture's `itemUsage[0].totalSample === skillUsage[0].totalSample === mercTypeUsage[0].totalSample`. If they diverge, the API's behavior changed upstream — flag.

**2. Item coverage ≥ 95%**
- Load `data/item-slots.json` (or whatever the app uses).
- Run our `shapeTopItemsBySlot(fixture.itemUsage)`.
- Count how many fixture rows ended up in any slot (sum of all slots' lengths).
- Assert `mappedCount / fixture.itemUsage.length >= 0.95`.
- This would have caught 2.1.2 (32% drop) immediately.

**3. No invented items**
- For every item in our output across all slots, assert it appears in `fixture.itemUsage`.
- We should never display an item the API didn't return.

**4. Slot consistency**
- For items we have in `item-slots.json`, our `shapeTopItemsBySlot` must put them in the slot the lookup table says.
- Pure-function check, fast.

**5. Percentage preservation**
- For each slot, sum the `pct` of items in our output.
- Sum the `pct` of fixture rows that mapped to that slot (using `slotFromItemName`).
- Assert the two are equal (within floating-point tolerance — 0.01).

### Not in this test file (deferred)

`aggregateSkillUsage`, `aggregateAffixModsBySlot`, `aggregateCharms`, `aggregateAvgStats` all operate on raw character objects, not server-aggregated endpoints. To validate them we'd need to capture raw character fixtures (~1 MB per build) and decide on a stable subset of characters. Defer to a follow-up sprint — see the Open Questions section.

### Tolerances

| Check | Threshold | Why |
|---|---|---|
| Item coverage | ≥ 95% mapped | Small slack for new PD2 items we haven't seen yet |
| No invented items | exactly 0 | We should never make up data |
| Slot consistency | exactly 100% | Pure-function check, no slack |
| Percentage preservation | within 0.01 absolute | Floating-point |
| totalSample consistency | exact match | API guarantee |

### What this catches and what it misses

**Catches:** every variant of "our display loses data from the API" (2.1.2), any change in our shaping logic that double-counts or drops items, slot-mapping regressions, API behavior changes that break our parsing.

**Misses:** any bug that's *upstream* of our app — e.g., us sending the wrong query params to the API (2.1.1). The refresh script uses our own API client, so a 2.1.1-shaped bug would appear in BOTH the production code AND the fixtures.

**Mitigation for the "misses" gap:** in the same test file, add a separate test class **API client URL contract** that:
- Doesn't hit the network.
- Mocks `global.fetch` and asserts the URL our `getItemUsage(filter, skills)` builds includes `skills=...` query param when called with non-empty skills.
- One test per endpoint, ~15 lines total.

This is the deepest fix for the 2.1.1 shape — even without live API access, we assert our URL is correct.

### CI integration

- Tests run as part of existing `npm test` — no new infra.
- Fixtures committed to the repo (small — ~50 KB per build × 7 = ~350 KB).
- CI fails on any assertion failure.
- On fixture refresh, if any test newly fails, that's the signal that something drifted upstream (API change) or something broke in our code.

## Bucket C — Integration-prep doc

New file: `plan/architecture/integration-notes.md`. Purpose: be the **agenda** for the conversation with coleestrin about how a port would actually work. Not a port plan, a "things to align on before porting" plan.

### Sections

1. **Current stack** (one paragraph) — Next.js 16 App Router, static export, zero backend, IndexedDB cache, web-worker aggregation, Tailwind/shadcn UI.

2. **What's in scope to port**:
   - The aggregation logic (`src/lib/aggregate/*`, `src/lib/shape/*`, `src/lib/filter.ts`, `src/lib/diff.ts`) — pure functions, no UI assumptions, trivially portable.
   - The data files (`data/skill-prereqs.json`, `data/item-slots.json`, `data/builds.json`) — domain knowledge, framework-agnostic.
   - The UI components (`src/components/*`) — React + Tailwind + shadcn. Portable IF their stack is also React-based.

3. **What changes in their environment**:
   - **Direct DB access** instead of public API. Solves "rares not aggregated" (server-side endpoint limitation). Solves the 2.1.1 cohort-mismatch class entirely — no URL-building bugs possible when querying SQL directly.
   - **No more snapshot fallback / IndexedDB cache** — their backend has the data live.
   - **No more pagination stitching** — single-query SQL.
   - **Auth / user context** if their site has it.

4. **Open questions for coleestrin** — flat list:
   - What framework? React? Same stack as our UI?
   - Sub-route or modal — `pd2.tools/aggregator`, `pd2.tools/builds/aggregate`?
   - Existing UI patterns to follow (their character page style)?
   - Does pd2.tools have a build-aggregator concept already, or are we adding a new one?
   - DB schema — what shape do characters / items live in on their side?
   - Test infra — do they have one we should plug into?
   - Style — adopt their CSS or contribute ours?

5. **Suggested next step** — when these questions are answered, decompose porting into Phase 2.x sprints (likely 2–4 weeks of work depending on stack overlap).

This doc lives in `plan/architecture/` because it's an architectural decision artifact, not a one-off note.

## Open questions

- **Item-slots auto-build on fixture refresh?** When the refresh script pulls new API data, should it also auto-run `build-item-slots-from-wiki.ts` to make sure the slot map has any new items? Or keep that manual? **Decision: keep manual.** Coupling them invites confusing failure modes; better to make fixture refresh single-purpose.
- **Should fixtures include character-level raw data?** Currently we use server-aggregated stats endpoints in tests. The client-side aggregator (`aggregateSkillUsage`, `aggregateAffixModsBySlot`, etc.) works on raw character data which the fixtures don't capture. **Decision: defer to Sprint 2.3.** Raw character fixtures would balloon fixture size (~1 MB per build) and require deciding on a stable subset of characters to capture. Out of scope here.
- **Notify coleestrin we're treating his offer as deferred + preparing?** **Decision: yes, as part of Sprint 2.2 close** — short courtesy reply once Bucket B + C are done. Out of code scope, in process scope.

- Porting any code to pd2-tools.
