# Sprint 2.3 — Meta integration into pd2.tools

**Date:** 2026-05-11
**Owner:** Steven Obst
**Status:** Approved (design)

## Context

After Sprint 2.2 closed, Steven exchanged Discord messages with [@coleestrin](https://github.com/coleestrin) (LAMP on Discord), the pd2.tools maintainer. Decision reached:

- Integration is happening — **not deferred**.
- New top-level menu entry on pd2.tools — `/meta` (or similar) — separate from `/builds`.
- Adopt pd2.tools' look and feel; no external standalone version long-term.
- Steven ships one cohesive PR when ready; coleestrin reviews/merges.

This sprint executes that integration. The standalone (`pd2-aggregator.vercel.app`) stays live during development as a reference implementation and parity baseline.

## Goals

1. **Working `/meta` page on pd2.tools** that delivers feature-parity with the standalone for: cohort filter (class + skills + game mode + min level), top items by slot, affix frequency, charm patterns, build sheet, mercenary, level distribution, build presets, and "Diff my character" mode.
2. **Direct Postgres aggregations** — replace browser → public API with browser → React Query → Express → SQL. Eliminates the 2.1.1 cohort-mismatch bug class entirely. Unlocks rare/magic/crafted item aggregation (the public API can't do this — direct SQL can).
3. **Match pd2.tools' stack and conventions** — Vite + React Router v7 + Mantine v7 + React Query + raw `pg` queries. ESLint clean, tsc clean, their CI green.
4. **Backend-side validation suite** — the parity tests from Sprint 2.2 reborn as Jest + supertest integration tests in `api/src/routes/meta.test.ts`. Same 7 canonical builds, same assertions, but testing the full SQL → API → JSON pipeline.
5. **One cohesive PR** to `coleestrin/pd2-tools` — fork-based workflow, develop in silence until checkpoint 8, then submit.

## Non-goals (scope-control during this sprint)

- **No vitest setup for pd2-tools' frontend.** They run only `tsc --noEmit + eslint + vite build` on FE. Adding a test runner is feature-creep on this PR. (See Deferred Follow-ups.)
- **No refactoring outside `/meta`-related files in pd2-tools.** Contributing, not redecorating. (See Deferred Follow-ups.)
- **No new product features beyond standalone parity.** Ship parity first. (See Deferred Follow-ups.)
- **No dual-path (public API + direct DB) in the fork.** Direct DB only. (Permanent constraint — not deferred.)
- **No sunsetting the standalone during development.** Keep `pd2-aggregator.vercel.app` running as the reference baseline until the PR merges.

### Validation strategy through the port

The Sprint 2.2 parity test suite (`PD2/src/lib/validation/parity.test.ts`, 42 sub-tests) moves to the fork as `api/src/routes/meta.test.ts`. Same 7 canonical builds (Hammerdin, Lightning Fury, etc.), same 5 assertion types per build. Difference:

- **Before:** test loads a captured JSON fixture and runs our `shapeTopItemsBySlot` against it
- **After:** test starts an in-memory Express app with the meta route mounted, hits `/api/meta` with the filter, asserts the response matches expected shape and that all items are slot-mapped, etc.

This is **strictly stronger** than the standalone version because it now also exercises the SQL → JSON pipeline. The fixture files become Jest test fixtures (or are replaced by SQL seed scripts in the test DB — TBD during implementation).

**Bonus**: a new test class — **standalone-vs-fork parity** — runs locally (not in CI) and asserts the fork's `/api/meta` produces the same output as the standalone's client-side aggregation for the same filter. This is the safety net during checkpoint development.
